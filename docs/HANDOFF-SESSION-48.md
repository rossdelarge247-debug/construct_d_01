## What could improve

- **Two verify-before-planning slips in one session.** Stage-2 surfaced two cases where kickoff facts didn't match live source (HANDOFF-47 row-8 + 6-AC shape locations). Both caught at draft time, but only because grep-first-before-Read happened; if I'd Read the cited location directly I'd have built on the wrong premise silently. Rule already exists in CLAUDE.md §Planning conduct ("Verify before planning" + "Distrust your own summaries") — the lesson is enforcement-by-habit isn't enough; cited locations should always be `grep -nc <heading-substring>`-verified before treating the citation as authoritative.
- **Stage-4 bigger slip: assumed PR #30 had merged because `HEAD == origin/main == 1e1f113`.** That equality only confirms branch alignment, not which PRs merged into that SHA. Session-48 pre-flight checklist (SESSION-CONTEXT L213) explicitly named `mcp__github__list_pull_requests state=closed perPage=3` as a turn-0 step; I skipped it. Cost: PR #32's claims about persona-suite-liveness were authored on a false premise; fix-up commit + verification.md §"Verify-before-planning slip" honest disclosure required. Hard rule for v3c carry-over: turn-0 verification step needs to be explicit, not implied by `git rev-parse`.
- **Rubric over-strictness on doc-only / deferred-slice / spec-design PRs.** Slice-reviewer persona's rubric appears to have one scoring path that conflates three distinct categories: (a) scope-creep on active-AC slice (caught correctly on PR #33 round 1), (b) scope-marker update on deferred-AC slice (likely PR #34's `block`), (c) spec-design content vs slice-impl content (likely PR #33's round 2 `block` post-revert). Without check-run output text retrievable via mcp tools, this is pattern-guess; but the diff-content profile post-revert is structurally inconsistent with category (a). v3c rubric extension needs separate categories.
- **Check-run output bodies not retrievable via mcp toolkit.** `mcp__github__pull_request_read get_check_runs` returns conclusion only; not `output.summary` / `output.text`. Public GitHub API rate-limited unauthenticated. `gh` CLI not available per harness instructions. User had to paste check-run summary manually for PR #33 round 1 finding. v3c carry-over: surface a tool path (mcp extension or auth-token-via-Bash) for retrieving check-run output bodies.
- **Three rounds of recursive findings before stepping back.** Smell-trigger rule was applied at round 3, which is the threshold the rule defines. No false positive (the rule fired correctly), but the iteration cost was real: ~30min of session time + 3 fix-up commits + 2 reverts. Pre-emptive trigger at round 2 would have been cheaper. v3c rubric should consider lowering the smell-trigger threshold for spec/doc PRs where iteration is structurally cheap (no impl regression risk) but rubric-mismatch findings are structurally costly (no real fix path on the PR's surface).
