- **Sub-slicing arithmetic check held.** α + β + γ + δ accounted for 100% of S-F7 scope per the table in acceptance.md. AC-arithmetic-check rule (lifted in session 31) earned its keep on first use.
- **Spec 72 §7 verbatim read promoted AC-3 from "recommended" to "required".** I had framed runtime assertion as a "cheap defence-in-depth nice-to-have" — re-reading the spec showed the wording is mandated. Quote-don't-paraphrase rule prevented an under-specified AC.
- **3 rule lifts pre-scaffolding paid off during slice docs.** Naming, function shape, and effects-behind-interfaces were explicit while drafting AC and test plans. Without them, the same principles would have been implicit and harder to reference in adversarial review later.
- **Behavioural-test discipline preserved.** All 6 test files exercise the interface with synthetic inputs, not file-content assertions. Per the rule lifted in session 31's wrap PR.
- **Hook surveillance worked twice.** stop-hook caught the untracked lockfile gap that would otherwise have lingered into session 33; line-count hook surfaced the 2000-line stop honestly even though most of the count was auto-generated noise.
- **Strategic conversation surfaced a real backlog-overview gap.** User wanted the multi-session forward map readable; the 10-slice table from kickoff served exactly that purpose. Worth keeping similar structure in future SESSION-CONTEXTs.

## What could improve

- **`pnpm install` should be a session-start verify item, not a mid-test-run discovery.** node_modules was missing at session start; install fired only when tests needed to run. Cost ~25s + a vitest 4.x reporter surprise that could have been caught earlier. Adding "if `tests/` exists, verify node_modules + lockfile" to session-start orientation would prevent this.
- **Lockfile gap caught at stop-hook, not at session start.** Same root cause as above. `pnpm-lock.yaml` was untracked despite not being gitignored — a latent project pattern that was easier to spot during install than at end-of-session under hook pressure.
- **Test files were generous on boilerplate.** ~481 lines for 6 RED files means avg ~80 lines/file. beforeEach/afterEach blocks repeat across files. Could be tighter via a shared setup helper (carry-forward as candidate for β when 6 more scenarios + dev surface tests land).
- **Strategic conversation took 6 turns before AC freeze.** Valuable, not wasted, but eats budget. Future kickoffs should include "is this still the right slice?" as an explicit pre-flight gate so the conversation is structurally legitimate rather than improvised.
- **Verification.md "consumer-smoke" pattern is novel for infrastructure slices.** The template assumes UI surface; α has none, so most rows are N/A. Worth a template variant for non-UI / library / contract slices. Engineering-phase-candidates entry.
- **Stop-hook line count vs authored line count diverged badly.** 5556-line lockfile + ~1100 authored = 6663 cumulative. Hook can't distinguish auto-generated from authored; correctly fired STOP, but the signal-to-noise was poor. Future: when committing large auto-generated artifacts (lockfiles, generated types), consider whether to gitignore or annotate to the hook somehow.

## Bugs / surprises

- **Vitest 4.x dropped `--reporter=basic`.** Works in 3.x; in 4.x it tries to load `basic` as a custom reporter module → `Failed to load url basic` startup error. Resolution: drop the flag (default reporter works) or use `--reporter=verbose` / `--reporter=dot`.
- **`pnpm-lock.yaml` previously untracked despite not being in `.gitignore`.** Project pattern unclear — committed as `4ee5334` for reproducibility; revert + gitignore if intentional. Bears verifying with the user before next session.
- **node_modules missing at session start** (harness clean state). Required mid-session `pnpm install`; ~25s + 1 unsuccessful test invocation before the install discovery.
- **2000-line hook STOP triggered by lockfile inflation.** 5556-line auto-generated lockfile + ~1100 authored = 6663 cumulative when the hook fired. Hook correctly applied the rule; the rule's signal was muddied. Worth flagging if it becomes a recurring pattern with future generated artifacts.
- **Compile-time RED is a weaker form of TDD red than runtime-RED.** Tests can't even import the units → vitest reports "Failed to resolve" rather than "assertion failed". Once impl ships, tests become runtime-RED (assertions execute and may still fail). Worth naming explicitly so reviewers don't mistake compile-RED for "tests just aren't wired" later.
