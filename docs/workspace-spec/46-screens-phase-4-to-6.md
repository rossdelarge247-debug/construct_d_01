# Spec 46 — Key Screens: Phases 4-6 (Agree, Finalise, Move On)

**Date:** 17 April 2026
**Purpose:** Screen-by-screen design for the second half of the journey — from "facts agreed" through proposal, agreement, consent order, submission, and implementation.

### S4.2 — Per-item proposal (the playbook engine)

```
┌───────────────────────────────────────────────────┐
│  The home                                         │
│  ─────────                                        │
│                                                    │
│  Net equity £230,000                              │
│                                                    │
│  What do you propose?                             │
│                                                    │
│  ○ I keep the house (buy Mark out — £115k to him) │
│  ● Sell and split 50/50                           │
│  ○ Sell and split — adjust [__]% to me            │
│  ○ Deferred sale when Jake turns 18               │
│  ○ Mark keeps the house (he buys me out)          │
│                                                    │
│  Why?                                             │
│  [We can't afford the mortgage on one income]     │
│                                                    │
│  ─── Context ───                                  │
│  • Net equity £230k → equal split = £115k each    │
│  • Your income £2,496/mo covers mortgage: no      │
│  • Mark's income £4,800/mo covers mortgage: yes   │
│  • Primary care: you (2 children, ages 4, 7)      │
│  • Typical range (needs-based): 50-60% to carer   │
│                                                    │
│  [Save and continue →]                            │
│                                                    │
└───────────────────────────────────────────────────┘
```

Same pattern for each major area. The **"Context"** section is the Juro playbook engine — arithmetic + typical range, never advice. Surfaces disclosed data relevant to the decision.

### S4.8 — Narrowing (system help)

When 1-2 items remain and the gap is small:

```
┌───────────────────────────────────────────────────┐
│  You're £18,000 apart                             │
│                                                    │
│  PENSIONS                                         │
│  Sarah: 0% share (£0)                             │
│  Mark: 40% share (£72k)                           │
│                                                    │
│  Ways to close this:                              │
│                                                    │
│  ○ Accept Mark's position (40% share)             │
│  ○ Offer a compromise — 20% share (£36k)          │
│  ○ Offset against another item                    │
│      Example: Mark takes 60% of joint savings     │
│      instead                                       │
│  ○ Get a mediator to help on this item            │
│                                                    │
│  💡 Private note: Your fallback was 20% share.    │
│  This would close the gap.                        │
│                                                    │
└───────────────────────────────────────────────────┘
```

The 💡 private note is the fallback-position reminder — only visible to the user who defined it.

## Summary: 30 critical screens

| Phase | Screens |
|---|---|
| 1. Start | Welcome, wellbeing check, 6 profiling screens, transition |
| 2. Build | Bank reveal, tier-1 matched, tier-2 disambiguation, tier-3 unknown, children, housing, future, spending, verification, document ready, document view |
| 3. Share | Ready to share, Mark's onboarding, reconciliation pass, unified view, item discussion, facts agreed |
| 4. Agree | Proposal intro, per-item proposal, children proposal, review before send, redline view, counter, progress board, narrowing, agreement |
| 5. Finalise | Generate, pre-flight, professional review, submit, waiting |
| 6. Move On | Sealed, implementation checklist, done |

Each screen follows the UX principles from spec 43. Each is designed to feel like editing or viewing THE document, not navigating an app.

## What this spec does NOT cover

- Mobile-specific adaptations (will need dedicated spec)
- Safeguarding-adapted flows (coercive control detection → alternative journey)
- Professional guest experience (mediator joining — future spec)
- Error handling and edge cases
- Specific visual design (that's a visual design pass, not a workflow spec)
